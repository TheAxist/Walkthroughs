The TCP Handshake: An Analysis

In the below examples, the client is recongized in Wireshark as having a 10.0.2.15 IP and the server with a 54.149.53.198 ip address. The Wireshark was ran from the clients perspective at the 10.0.2.15 IP address. NetCat was used to simulate network connections made outbound from the client to the server. 

To explain the IP addresses clearly: 

Client: 10.0.2.15       internal IP 

	98.190.145.152  external IP 

Server: 172.31.7.27     internal IP 

	54.149.153.189  external IP


Graceful Teardown (client closes connection)

Below the Client initiates a netcat session to the server.

Connection is successful. We observe the three way “TCP handshake” between our source IP and the Target Ip. The tcp handshake can be observed in the first 3 packets containing the red bracket. The following four packets (12,13,16,17) are the graceful tear down of the connection.



Graceful Teardown (server closes the connection)

This time, like before the client initiates the connection to the server, however now the Server terminates the connection.

Connection successful! Again we observe the three way handshake in the first three packets (52, 53, 54) in the form of a SYN > SYN, ACK > ACK. This time the server closes the connection sending a FIN, ACK, to which the client returns a FIN, ACK, and the server acknowledges with an ACK.

Fixed Code Snippet

Now for the fix. Below is a screenshot of the fixed php code snippet. “Solved.php”. We will now run it through the same four scenarios as above. The fix involves the addition of basename() and realpath(). As described in the PHP documentation, basename() will return the trailing name component, however is not aware of path components such as “..”. This is why realpath() is also necessary in conjunction with the “basename()” function. As described in the PHP documentation “realpath()” returns canonicalized absolute pathname .
